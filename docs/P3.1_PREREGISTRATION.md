## 0. What P3 is — and is NOT (council reframe)

Generalization / method-risk test on a non-Montecito fire (South Fork, 2024, NM). **P3 is primarily an *engineering* generalization test:** does the frozen pipeline produce **well-formed output** and **fail loudly** on different terrain, rather than depending on Montecito's geomorphology. The load-bearing gate is §6(a)+(b).

P3 is **NOT** a second validation event and does **not** produce a "VALIDATED" outcome. The reference anchors (§5) are a **clearly-secondary directional sniff-test**, interpreted cautiously and never reported as validation — P4 is the second validation event. Gate slogan unchanged: **sane ranking or loud failure, never silent bad output.**

## 5. Reference handling — FROZEN: directional sniff-test only, no metric

Two references, both **directional, secondary, never a published number** (per the §0 reframe):

1. **USGS hazard reference — anchor on combined-hazard CLASS (`H_*`) + Gartner volume (`V_*`), NOT the `P_*` likelihood.** `[COUNCIL refinement; adversarial-review precision]` *Why:* the M1 likelihood is dNBR-dominated via its `F` input (`F = mean dNBR/1000`), so comparing the dNBR arm to `P_*` is largely dNBR-vs-dNBR. `H_*`/`V_*` are **less** circular but **not clean** — and the residual matters, so locate it precisely: the genuinely dNBR-**independent** terms are **`S` (soil KF), `Relief`, basin `Area`, and the slope-≥23° geometry**. `T` ("proportion of catchment burned at **mod/high severity** AND slope ≥23°") retains a burn-severity → dNBR path through its severity half, and `V_*` scales with `Bmh_km2` (area burned mod/high), also dNBR-derived. So anchor on the clean terms (`S`/`Relief`/`Area`/slope-geometry); use `H_*`/`V_*` for coarse directional agreement only, knowing the residual confound lives in `T`'s severity half and `V`'s `Bmh`.
2. **Documented located flows — RESOLVED, n=2 (`[VERIFIED]`):** no USGS observation/inventory dataset covers these fires (the two existing inventories end 2023), so the list is the public USGS-captioned record. Within the South Fork AOI the documented debris-flow positives are exactly **two basins**: **`sfk2024` FID 70 = North Fork Cedar Creek** (3.43/3.43 km channel in-basin) and **FID 100 = South Fork Cedar Creek** (3.65/3.69 km, 99%). Bear Canyon is in Salt (`slt2024`), not South Fork; the Rio Ruidoso flash floods are flooding, not in-basin debris flow — both correctly excluded. **No surveyed no-flow set exists → only the 2 positives are labeled; the other 190 basins are unlabeled, not negatives.**

**Anchor version pin (`[VERIFIED]` flag; stated assumption):** use the staged **`sfk2024` v1.0** (emergency assessment 2024-07-01) as the frozen anchor — contemporaneous with the fire and the documented flows, and the version whose `F=dNBR/1000` structure underpins §5(1). A newer **v3.0** re-assessment (2026-05-29, PWFDF v1.1.0) exists; **do not switch to it** — chasing the newest model version is the oracle-tuning the firewall forbids, and the anchor is secondary regardless. (Optional later cross-check against v3.0 = non-gating diagnostic, not the frozen anchor.)

### Secondary — directional sniff-test (interpret cautiously, small-n)

**(c)** Spatial check. **All joins are spatial, not by FID index** — `[VERIFIED, corrected]` the reference has no basin-ID field, *and* basin↔outlet pairing within the USGS data is itself spatial, not positional (the earlier "positional FID 0..191" claim was tested and refuted: only ~13% of outlet[i] lie within 50 m of basin[i]); the tool also delineates its own basins, so there is no 1:1 map to USGS's 192. P3.4 anchor-join code must spatially link, never assume index alignment.
- The **2 documented-flow basins (FID 70, FID 100)** are predicted to fall in the **upper half** of the within-fire ranking — *a pre-registered prediction to test at P3.4, not an already-checked result.* *Caveats stated in the bar itself:* (i) **n=2, spatially clustered** (one creek, two forks) — a weak confirmatory signal, not validation; its failure is a **FLAG to investigate / finding-candidate, not an automatic FAIL** (§7); (ii) **area-term confound** — both basins are among the largest in the fire (area rank 2 and 3 of 192), so the `× area` term alone would rank them high regardless of burn/slope discrimination; passing this bar therefore does **not** isolate whether the *method* discriminates. ## 7. Outcome taxonomy — FROZEN (renamed; escape hatch closed)

P3 never yields "VALIDATED." Outcomes:

- **RAN-CLEAN — method transfers.** §6(a) holds, §6(b) guards verified, §6(c) sniff-test consistent. The load-bearing engineering gate carries this; the sniff-test is supporting, not validating.
- **RAN-CLEAN — finding (method boundary).** The pipeline halts **loudly** due to a **genuine geomorphic/structural property of the fire** the method legitimately can't handle — *traced to the fire, reproducible, not to a code defect* (e.g. no clean mountain-front → degenerate delineation). A documented finding, not a failure.
- **FAIL.** Either (i) **silent bad output** — guards did not fire when §6(b) requires (the primary, hardest-to-see failure), or (ii) a loud halt / misbehavior **traced to a code defect, not the fire** — i.e. a **BUG**, which is NOT a method-boundary finding: fix and rerun; a bug that yields silent wrong output is a FAIL.

**Escape-hatch rule (council):** a loud failure is a *finding* only if traced to the fire's geomorphology; if traced to implementation, it is a *bug* (fix, don't reclassify as a finding). The §6(c) sniff-test failing, given small-n, is a flag/finding-candidate — never on its own a FAIL.
