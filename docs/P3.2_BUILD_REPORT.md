## 2. Two items investigated and CLEARED (neither is a finding)

### 2a. Cross-WRS-path scene pair — NOT a problem
The frozen dates fall on adjacent Landsat-9 overpasses, so PRE is path 033 and POST is path 032 (the
AOI sits in the 032/033 sidelap) — unlike Thomas, which used one path (042/036) for both. The frozen
`p2_acquire_dnbr` forms `dNBR = NBR_pre − NBR_post` on the **identical native grid** and asserts
`tpre == tpost`. I verified from `proj:transform` metadata that both scenes share the common
Collection-2 30 m lattice (both UL corners ≡ 15 m mod 30) and **snap to the identical native window
over the AOI** (UL 426405.0, 3697305.0; 480×322). The subtraction is pixel-aligned and the frozen
guard passes. Recorded in the dNBR provenance as a *characteristic*, not a deviation.

### 2b. Cloud measurement basis — my acceptance check was wrong, the scene is sound
My first acceptance check measured pre-scene AOI cloud at **13.62%** over the **bbox AOI** and I treated
it as a freeze-gate STOP. That was a basis error on my side. The P3.1 freeze-gate's recorded **6.3%/0%**
were measured over the **fire perimeter**, and the frozen `_qa_cloud_mask` (bits 1,2,3,4) over the
perimeter reproduces them to **6.32% / 0.0%** ([VERIFIED 2026-06-20]). The excess that inflated the
bbox number is cloud (mostly shadow + cirrus) in the **AOI margins outside the burn** — below-floor,
non-covered cells (weight 0) that don't affect screening. I corrected the acquire to report both bases
and the acceptance check to compare against the **perimeter** value. Scene freeze **PASSES**.

> Carry-forward caveat (P3.5 / `limitations.md`): cloud is **NOT removed** from the product (Decision A).
> The in-perimeter 6.32% injects spurious dNBR into ~6% of burned cells (the raw dNBR min −1.285 reflects
> shadow/cloud artifacts). The full-AOI (bbox) cloud is higher (13.62%) but out-of-burn.

## 3. The blocker — frozen pipeline is CRS-locked to the validation zone

`src.ingest.ingest_dnbr_both_arms` aborts loudly:
```
GATE ABORT (fail-loud, A8): DEM CRS EPSG:32613 != EPSG:32611
```
- [`config.py:49`](../src/config.py#L49): `CANONICAL_CRS = "EPSG:32611"` (the Montecito/UTM-11N validation zone)
- [`grids.py:51-52`](../src/grids.py#L51-L52): `assert_aligned` raises `GateAbort` unless `ref_profile["crs"] == CANONICAL_CRS`
- [`ingest.py:257-258`](../src/ingest.py#L257-L258): the seam calls `assert_aligned` on both arms

**The tension:** A24 §3 designates the **grid/CRS as "the one legitimate per-fire change"** (EPSG:4269 →
EPSG:32613 for South Fork). But the frozen pipeline hardcodes the validation CRS and there is no per-fire
config mechanism (one `config.py`, shared with the behavior-lock gate that **must** stay 32611 to
reproduce AUC 0.987). So the build prompt's instruction to run `ingest_dnbr_both_arms` **unmodified** is
incompatible with its own per-fire CRS change. Per §5 ("`assert_aligned` GateAbort → STOP, report,
do not resample-to-force") and the Tier-1 HALT rule, I stopped rather than edit a frozen value.

Per the §7 taxonomy this is an **implementation generalization gap** (the pipeline was built CRS-locked
to the validation fire), surfaced early — exactly what P3 exists to catch. It is not geomorphic (not a
method-boundary finding) and not a silent failure (the guard fired correctly — that part is working).

## 4. Decision (my judgment) + justification

**Decision: stop P3.2 at acquisition; the CRS generalization is the first P3.3 task; do not modify any
frozen code/value now; commit nothing.**

Justification:
1. **It is P3.3 by A24 §8's own phasing** — "P3.3 = net-new fail-loud + boundary-assertion code, the
   heaviest, new code, no oracle." CRS generalization is precisely that. Folding it into P3.2 would
   conflate acquisition with generalization and skip the dedicated build+adversarial-review cycle A24 §8
   mandates for the riskiest chunk.
2. **The Montecito behavior-lock is too important to touch casually.** Any change to `CANONICAL_CRS` /
   `assert_aligned` must be proven not to regress the 32611 reproduction (AUC 0.987, 6/6 flowed in top
   tercile). That proof is a reviewed change, not a build-instance side effect.
3. **P3.2's real deliverables are done and sound** — acquisition + a passing scene freeze. The ingest
   step was mis-scoped into P3.2 by the prompt (it assumed a CRS-generality the frozen code lacks).
4. **The guard firing is correct behavior** (A8). Forcing past it (resample-to-32611, or globally
   editing the literal) is exactly the silent-degradation the firewall forbids.

## 5. Future recommendations (for P3.3)

**Primary: make the canonical CRS per-fire, additively (default preserves 32611).**
- Thread an optional `canonical_crs` parameter through `ingest_dnbr_both_arms → reproject_dnbr →
  assert_aligned`, **defaulting to `config.CANONICAL_CRS` (32611)** so the gate/Montecito path is
  byte-for-byte unchanged; pass `"EPSG:32613"` for South Fork. Smallest frozen-code surface, provably
  non-regressing, easy to adversarially review. This matches the architecture's stated intent
  (`config.py` = "per-fire scalar tunables … stored per fire; never globally edited in place").
- **Regression gate:** after the change, re-run the behavior lock / `gate.py` and confirm the 32611
  path is unchanged (AUC 0.987, basin set, 39.19 km² master outlet) **before** re-running South Fork.

**Design question for the P3.3 review (don't decide unilaterally):** `assert_aligned`'s actual job is to
confirm two layers share **one grid** — arguably it should assert `ref.crs == other.crs` (agreement),
not `ref.crs == a fixed zone`. That is the more general guard, but it's a semantics change to a frozen
boundary check and could weaken the Montecito guard, so it belongs in front of the council, not in a
build instance. The additive-parameter approach above is the lower-risk default.

**Also CRS-touched downstream (audit during P3.3, not yet hit):**
- [`outputs.py:87`](../src/outputs.py#L87) builds the output GeoDataFrame with `crs=CANONICAL_CRS` before
  `.to_crs(4326)` — needs the per-fire CRS at the P3.4/P3.5 output stage.
- [`grids.py:31`](../src/grids.py#L31) `centre_xy` docstring names EPSG:32611; verify its logic is
  transform-based / CRS-agnostic (it appears to be) so it needs no change.

**Carry-forward caveats (P3.5 / `limitations.md`):** (i) cloud-not-removed (§2b); (ii) the freeze-gate's
6.3% is a **perimeter** measurement — full-AOI cloud is higher but out-of-burn; (iii) the dNBR
